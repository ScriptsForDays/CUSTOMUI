Get started with WindUI Custom

Quick Start

Load WindUI Custom directly from GitHub:

local WindUI = loadstring(game:HttpGet('https://raw.githubusercontent.com/ScriptsForDays/CUSTOMUI/refs/heads/main/dist/main.lua'))()

Basic Usage

After loading WindUI, create a window and start adding elements:

-- Load WindUI
local WindUI = loadstring(game:HttpGet('https://raw.githubusercontent.com/ScriptsForDays/CUSTOMUI/refs/heads/main/dist/main.lua'))()

-- Create a window
local Window = WindUI:CreateWindow({
    Title = "My Script",
    Author = "by YourName",
    Folder = "myscript",
    Icon = "settings"
})

-- Create a tab
local Tab = Window:Tab({
    Title = "Main",
    Icon = "home"
})

-- Add a button
Tab:Button({
    Title = "Click Me",
    Icon = "mouse-pointer-click",
    Callback = function()
        print("Button clicked!")
    end
})

Key System

WindUI includes a built-in key system to protect your scripts. You can add it when creating a window by including the KeySystem parameter in your configuration.

Basic Key System

The simplest way to add a key system is with a single key or multiple keys:

-- Single key
local Window = WindUI:CreateWindow({
    Title = "My Script",
    Author = "by YourName",
    Folder = "myscript",
    Icon = "settings",
    KeySystem = {
        Key = "YOUR-KEY-HERE",
        SaveKey = true, -- Save key locally (optional)
        Note = "Enter your key to access the script" -- Optional note text
    }
})

-- Multiple keys (array)
local Window = WindUI:CreateWindow({
    Title = "My Script",
    Author = "by YourName",
    Folder = "myscript",
    Icon = "settings",
    KeySystem = {
        Key = {"KEY-1", "KEY-2", "KEY-3"}, -- Multiple valid keys
        SaveKey = true
    }
})

Key System with URL

You can add a URL where users can get their key (Discord, Linkvertise, Pastebin, etc.):

local Window = WindUI:CreateWindow({
    Title = "My Script",
    Author = "by YourName",
    Folder = "myscript",
    Icon = "settings",
    KeySystem = {
        Key = "YOUR-KEY-HERE",
        URL = "https://discord.gg/yourserver", -- Link to get key
        SaveKey = true,
        Note = "Join our Discord to get your key!"
    }
})

Key System with Custom Folder

Specify a custom folder name for saving keys. The folder is created at the root level (not inside WindUI):

local Window = WindUI:CreateWindow({
    Title = "My Script",
    Author = "by YourName",
    Folder = "myscript",
    Icon = "settings",
    KeySystem = {
        Key = "YOUR-KEY-HERE",
        SaveKey = true,
        KeyFolder = "MyKeyFolder" -- Custom folder name (optional, defaults to Config.Folder)
    }
})

Note: If KeyFolder is not specified, it defaults to Config.Folder (or "WindUIKeys" if neither is provided). The key file is saved as {KeyFolder}/{hwid}.key at the root level.

Key System with Discord Button

Add a Discord button next to the Exit button that copies your Discord link to clipboard:

-- Simple: Just provide the Discord URL string
local Window = WindUI:CreateWindow({
    Title = "My Script",
    Author = "by YourName",
    Folder = "myscript",
    Icon = "settings",
    KeySystem = {
        Key = "YOUR-KEY-HERE",
        Discord = "https://discord.gg/yourserver", -- Discord button enabled with this URL
        SaveKey = true
    }
})

-- Advanced: Control with table
local Window = WindUI:CreateWindow({
    Title = "My Script",
    Author = "by YourName",
    Folder = "myscript",
    Icon = "settings",
    KeySystem = {
        Key = "YOUR-KEY-HERE",
        Discord = {
            Enabled = true, -- Enable/disable Discord button
            URL = "https://discord.gg/yourserver" -- Discord invite link
        },
        SaveKey = true
    }
})

Note: The Discord button appears next to the Exit button with a "key" icon (same as the URL key system). When clicked, it copies the Discord URL to clipboard and shows a notification.

API-Based Key System

WindUI supports integration with key system services like PlatoBoost, Panda Development, and Luarmor:

local Window = WindUI:CreateWindow({
    Title = "My Script",
    Author = "by YourName",
    Folder = "myscript",
    Icon = "settings",
    KeySystem = {
        Note = "Enter your key to access the script",
        SaveKey = true,
        API = {
            -- PlatoBoost
            {
                Type = "platoboost",
                ServiceId = 1234,
                Secret = "your-secret-key"
            },
            -- Panda Development
            {
                Type = "pandadevelopment",
                ServiceId = "your-service-id"
            },
            -- Luarmor
            {
                Type = "luarmor",
                ServiceId = "your-service-id"
            }
        }
    }
})

Key System Parameters

Parameter Type Description
Key string/table Valid key(s). Can be a single string or table of strings for multiple keys.
SaveKey boolean Automatically save and load the key locally (default: false). If true, users won't need to enter the key again.
Note string Optional note text displayed in the key system dialog.
URL string Link where users can get their key (Discord, Linkvertise, Pastebin, etc.).
Thumbnail table Custom thumbnail image. Contains Image (string), Title (string, optional), and Width (number, optional, default: 200).
API table Array of API service configurations. Supported types: "platoboost", "pandadevelopment", "luarmor".
Discord string/table Discord button configuration. Can be a string (Discord URL) or table with Enabled (boolean) and URL (string). When enabled, adds a Discord button next to the Exit button that copies the Discord link to clipboard. The Discord button uses the same "key" icon as the URL key system.
KeyFolder string Custom folder name for saving keys. If not specified, defaults to Config.Folder (or "WindUIKeys" if neither is provided). The folder is created at the root level (not inside WindUI). The key file is saved as {KeyFolder}/{hwid}.key.

How It Works

  • When KeySystem is provided, the window will not load until a valid key is entered
  • If SaveKey is true, the key is saved locally and automatically loaded on next run
  • If KeyFolder is specified, keys are saved in that custom folder at the root level (not inside WindUI)
  • If KeyFolder is not specified, it defaults to Config.Folder (or "WindUIKeys" if neither is provided)
  • If API is provided, keys are validated through the specified service(s)
  • If URL is provided, users can click to open the link and get their key
  • The key system dialog appears before the main window, blocking access until a valid key is entered
  • All copy actions (Discord, URL, API) show notifications when links are copied to clipboard

Config System

WindUI includes a powerful config system that allows users to save and load their settings, including all UI element values and custom theme colors. Configs are saved locally and can be named for easy management.

Enabling the Config System

To enable the config system, simply provide a Folder parameter when creating your window:

local Window = WindUI:CreateWindow({
    Title = "My Script",
    Author = "by YourName",
    Folder = "myscript", -- Config system enabled with this folder name
    Icon = "settings"
})

Basic Config Usage

Once the config system is enabled, you can access it through Window.ConfigManager:

local ConfigManager = Window.ConfigManager

-- ⚠️ IMPORTANT: Always use Flag parameter for toggles, dropdowns, etc.
-- Elements with Flag are automatically registered and their states are saved/loaded
local myToggle = Tab:Toggle({
    Title = "Enable Feature",
    Default = false,
    Flag = "featureToggle", -- Required! Without this, state won't be saved
    Callback = function(value)
        print("Feature enabled:", value)
    end
})

local myDropdown = Tab:Dropdown({
    Title = "Select Option",
    Values = {{Title = "Option 1"}, {Title = "Option 2"}},
    Flag = "optionDropdown", -- Required! Without this, selection won't be saved
    Callback = function(option)
        print("Selected:", option.Title)
    end
})

-- Create a config with a custom name
local configName = "myconfig"
local configFile = ConfigManager:CreateConfig(configName)

-- Note: Elements with Flag are automatically registered - no manual Register() needed!
-- But you can still manually register if needed:
-- configFile:Register("featureToggle", myToggle)

-- Save custom data (optional)
configFile:Set("playerName", "Player1")
configFile:Set("lastSave", os.date())

-- Save the config (returns success, errorMessage)
-- ✅ Toggle on/off state and dropdown selection are automatically saved!
local success, errorMsg = configFile:Save()
if success then
    print("Config saved successfully!")
else
    warn("Failed to save config:", errorMsg)
end

-- Load the config
-- ✅ Toggle and dropdown states are automatically restored!
local loadSuccess, loadResult = ConfigManager:LoadConfigByName("myconfig")
if not loadSuccess then
    warn("Failed to load config:", loadResult)
end

⚠️ Common Mistakes to Avoid

Important: The config system does NOT use methods directly on the Window object. These methods do not exist and will cause errors:

-- ❌ WRONG - These methods do NOT exist on Window:
Window:Save("myconfig") -- This will fail!
Window:Load("myconfig") -- This will fail!
Window:GetConfigs() -- This will fail!
Window:Delete("myconfig") -- This will fail!

✅ Correct Usage: Always use Window.ConfigManager to access config functionality:

local ConfigManager = Window.ConfigManager

-- ✅ CORRECT - Get all configs
local allConfigs = ConfigManager:AllConfigs()

-- ✅ CORRECT - Save a config
local configFile = ConfigManager:CreateConfig("myconfig")
local success, errorMsg = configFile:Save()

-- ✅ CORRECT - Load a config by name
local success, result = ConfigManager:LoadConfigByName("myconfig")

-- ✅ CORRECT - Delete a config
local success, result = ConfigManager:DeleteConfig("myconfig")

-- ✅ CORRECT - Check if config exists
if ConfigManager:ConfigExists("myconfig") then
    print("Config exists!")
end

Quick Reference: ConfigManager Methods

Method Description Returns
ConfigManager:AllConfigs() Get array of all saved config names Array of strings
ConfigManager:CreateConfig(name) Create or get existing config by name ConfigModule object
ConfigManager:LoadConfigByName(name) Load a config by its name success (bool), result (string/error)
ConfigManager:DeleteConfig(name) Delete a config by name success (bool), result (string/error)
ConfigManager:ConfigExists(name) Check if a config exists Boolean
configFile:Save() Save the config (called on ConfigModule) success (bool), errorMsg (string)
configFile:Load() Load the config (called on ConfigModule) success (bool), result (string/error)

Naming and Managing Multiple Configs

Users can create multiple named configs and easily switch between them:

local ConfigManager = Window.ConfigManager
local configName = "default"

-- Get all available configs
local allConfigs = ConfigManager:AllConfigs()
print("Available configs:", table.concat(allConfigs, ", "))

-- Create or get existing config by name
local configFile = ConfigManager:CreateConfig(configName)

-- Save the config (returns success, errorMessage)
local success, errorMsg = configFile:Save()
if success then
    print("Config saved successfully!")
else
    warn("Failed to save config:", errorMsg)
end

-- Load a config by name
local success, result = ConfigManager:LoadConfigByName("myconfig")
if success then
    print("Config loaded successfully!")
else
    warn("Failed to load config:", result)
end

-- Check if a config exists
if ConfigManager:ConfigExists("myconfig") then
    print("Config exists!")
end

Config Overwriting Behavior

Important: When you save a config with a name that already exists, it will automatically overwrite the existing config file. This means:

  • No confirmation needed: Saving a config with an existing name immediately replaces the old config
  • All data is replaced: The old config's UI element values, custom data, and theme colors are completely replaced with the new values
  • No backup created: The previous config data is not backed up - make sure you want to overwrite before saving
-- Example: Overwriting a config
local ConfigManager = Window.ConfigManager

-- Save a config named "myconfig"
local config1 = ConfigManager:CreateConfig("myconfig")
config1:Set("setting1", "value1")
config1:Save()

-- Later, save another config with the same name
-- This will OVERWRITE the previous "myconfig"
local config2 = ConfigManager:CreateConfig("myconfig")
config2:Set("setting1", "value2") -- Different value
config2:Set("setting2", "new") -- New setting
config2:Save()
-- ✅ The old "myconfig" is now replaced with the new data
-- ✅ setting1 is now "value2" (not "value1")
-- ✅ setting2 is now "new"

Properly Overwriting an Existing Config

Important: When overwriting an existing config, always use CreateConfig() instead of GetConfig(). Here's why:

  • GetConfig() only works if the config is already in memory: It returns nil if the config file exists but hasn't been loaded yet
  • CreateConfig() always works: It gets the existing config if it's in memory, or creates a new config module if it's not
  • CreateConfig() sets the config as current: This ensures all pending UI elements are properly registered
-- ❌ WRONG - Don't use GetConfig() for overwriting
local configFile = ConfigManager:GetConfig("myconfig")
if not configFile then
    warn("Config not found!") -- This will fail if config isn't in memory
    return
end
configFile:Save() -- May not work properly

-- ✅ CORRECT - Use CreateConfig() for overwriting
local configFile = ConfigManager:CreateConfig("myconfig")
-- Register all UI elements (toggles, dropdowns, etc.)
if Window.PendingFlags then
    for flag, element in pairs(Window.PendingFlags) do
        configFile:Register(flag, element)
    end
end
-- Register any additional elements you've stored
for flagName, element in pairs(MyStoredElements) do
    if element and element.__type then
        configFile:Register(flagName, element)
    end
end
-- Save the config (overwrites existing file)
local success, errorMsg = configFile:Save()
if success then
    print("Config overwritten successfully!")
else
    warn("Failed to overwrite config:", errorMsg)
end

Key Points:

  • Always use CreateConfig() when overwriting - it works whether the config is in memory or not
  • Register all UI elements before saving to ensure their current states are saved
  • Register both Window.PendingFlags (auto-registered elements) and any manually stored elements
  • The Save() method automatically overwrites the existing config file

Complete Example: Config UI

Here's a complete example showing how to create a config management UI with both dropdown and textbox options for loading configs:

local ConfigTab = Window:Tab({
    Title = "Config",
    Icon = "folder"
})

local ConfigManager = Window.ConfigManager
local saveConfigName = "default"
local loadConfigName = "default"

-- Input field for naming configs to save
local saveNameInput = ConfigTab:Input({
    Title = "Config Name (Save)",
    Desc = "Enter a name for your config",
    Value = saveConfigName,
    Callback = function(value)
        saveConfigName = value or "default"
    end
})

-- Dropdown to see all available configs (optional - for loading)
local configDropdown
local selectedConfigFromDropdown = nil
local function UpdateConfigDropdown()
    local allConfigs = ConfigManager:AllConfigs()
    
    -- Convert array of strings to dropdown format
    local dropdownValues = {}
    for _, configName in ipairs(allConfigs) do
        table.insert(dropdownValues, {
            Title = configName,
            Icon = "file"
        })
    end
    
    if configDropdown then
        configDropdown:SetValues(dropdownValues)
    else
        configDropdown = ConfigTab:Dropdown({
            Title = "Select Config (Optional)",
            Desc = "Choose a config from the dropdown, or type one below",
            Values = dropdownValues,
            Callback = function(option)
                selectedConfigFromDropdown = option and option.Title or nil
                if selectedConfigFromDropdown then
                    loadNameInput:Set(selectedConfigFromDropdown)
                    loadConfigName = selectedConfigFromDropdown
                end
            end
        })
    end
end

-- Initialize dropdown
UpdateConfigDropdown()

-- Input field for loading configs by name (optional - user can type here)
local loadNameInput = ConfigTab:Input({
    Title = "Load Config by Name",
    Desc = "Type the name of the config to load, or select from dropdown above",
    Value = loadConfigName,
    Callback = function(value)
        loadConfigName = value or ""
        -- Clear dropdown selection when user types manually
        selectedConfigFromDropdown = nil
    end
})

-- Save Config Button
ConfigTab:Button({
    Title = "Save Config",
    Icon = "save",
    Justify = "Center",
    Callback = function()
        if not saveConfigName or saveConfigName == "" then
            WindUI:Notify({
                Title = "Error",
                Desc = "Please enter a config name",
                Icon = "x",
                Duration = 3
            })
            return
        end

        -- Create or get existing config (will overwrite if exists)
        Window.CurrentConfig = ConfigManager:CreateConfig(saveConfigName)
        
        -- Save returns (success, errorMessage)
        local success, errorMsg = Window.CurrentConfig:Save()
        if success then
            WindUI:Notify({
                Title = "Config Saved",
                Desc = "Config '" .. saveConfigName .. "' saved successfully",
                Icon = "check",
                Duration = 3
            })
            -- Update dropdown to show new config
            UpdateConfigDropdown()
        else
            WindUI:Notify({
                Title = "Error",
                Desc = errorMsg or "Failed to save config",
                Icon = "x",
                Duration = 3
            })
        end
    end
})

-- Load Config Button (uses dropdown selection if available, otherwise uses textbox)
ConfigTab:Button({
    Title = "Load Config",
    Icon = "refresh-cw",
    Justify = "Center",
    Callback = function()
        -- Priority: Use dropdown selection if available, otherwise use textbox value
        local configToLoad = selectedConfigFromDropdown or loadConfigName
        
        if not configToLoad or configToLoad == "" then
            WindUI:Notify({
                Title = "Error",
                Desc = "Please select a config from the dropdown or enter a config name",
                Icon = "x",
                Duration = 3
            })
            return
        end

        -- LoadConfigByName returns (success, result)
        local success, result = ConfigManager:LoadConfigByName(configToLoad)
        if success then
            WindUI:Notify({
                Title = "Config Loaded",
                Desc = "Config '" .. configToLoad .. "' loaded successfully",
                Icon = "check",
                Duration = 3
            })
            -- Update save name to match loaded config
            saveConfigName = configToLoad
            saveNameInput:Set(saveConfigName)
        else
            WindUI:Notify({
                Title = "Error",
                Desc = result or "Failed to load config",
                Icon = "x",
                Duration = 3
            })
        end
    end
})

How the Config Loading Works

The config loading UI provides two flexible options for users:

  • Dropdown Selection: Users can select a config from the dropdown menu, which automatically lists all saved configs. When a config is selected, it updates the textbox below.
  • Textbox Entry: Users can manually type a config name in the textbox. This works independently from the dropdown and is useful when users know the exact config name.
  • Load Button: The load button intelligently uses the dropdown selection if available (priority), otherwise it uses the textbox value. This gives users flexibility to use either method.

Note: Both methods are optional - users can use either the dropdown, the textbox, or both. The dropdown automatically updates when new configs are saved.

Automatic Theme Color Saving

All custom theme colors are automatically saved and loaded with configs! You don't need to manually register or save them - the config system handles everything automatically.

What Gets Saved Automatically

When you save a config, the system automatically captures all custom theme colors you've set using any SetCustom* method. This includes:

  • Basic Properties: Accent, Background, Text, Button, Icon, Dialog, Outline, Hover, Placeholder
  • Window Properties: WindowTopbarTitle, WindowTopbarAuthor, WindowTopbarIcon, WindowTopbarButtonIcon, WindowTopbarBackground, WindowBackground
  • Tab Properties: TabBackground, TabTitle, TabIcon
  • Element Properties: ElementBackground, ElementTitle, ElementDesc, ElementIcon
  • Popup Properties: PopupBackground, PopupTitle, PopupContent, PopupIcon
  • Dialog Properties: DialogBackground, DialogTitle, DialogContent, DialogIcon
  • Component Properties: Toggle, Checkbox, CheckboxIcon, DropdownSelected
  • Any Custom Property: Properties set with SetCustomProperty()

How It Works

The config system automatically:

  1. On Save: Scans all custom theme overrides in Creator.CustomOverrides and serializes them (supports both Color3 and Gradient types)
  2. On Load: Restores all saved custom colors and applies them to the UI immediately
  3. No Manual Work: You don't need to call any special methods or register colors - it's completely automatic
-- Set custom colors using any SetCustom method
WindUI:SetCustomAccent(Color3.fromHex("#30ff6a"))
WindUI:SetCustomDropdownSelected(Color3.fromHex("#60a5fa"))
WindUI:SetCustomBackground(Color3.fromHex("#1a1a1a"))
WindUI:SetCustomText(Color3.fromHex("#ffffff"))
WindUI:SetCustomButton(Color3.fromHex("#52525b"))

-- You can also use gradients
WindUI:SetCustomBackground(WindUI:Gradient({
    ["0"] = { Color = Color3.fromHex("#0f0c29"), Transparency = 0 },
    ["100"] = { Color = Color3.fromHex("#302b63"), Transparency = 0.1 },
}, {
    Rotation = 45,
}))

-- Save config - ALL custom colors are automatically included!
local ConfigManager = Window.ConfigManager
local configFile = ConfigManager:CreateConfig("myconfig")

-- Register UI elements (optional - only if you want to save element values)
configFile:Register("myToggle", myToggle)
configFile:Register("mySlider", mySlider)

-- Save - custom colors are automatically saved, no extra code needed!
configFile:Save()

-- Later, load the config - ALL custom colors are automatically restored!
ConfigManager:LoadConfigByName("myconfig")
-- All your custom colors are now applied to the UI automatically

Important Notes

  • No Registration Required: Custom theme colors don't need to be registered with configFile:Register() - they're saved automatically
  • Supports All Color Types: Both Color3 and Gradient colors are fully supported and properly serialized/deserialized
  • Immediate Application: When you load a config, all custom colors are immediately applied to the UI - no manual refresh needed
  • Complete Persistence: All custom theme overrides are preserved, so users can have different color schemes for different configs
  • Works with All Methods: Any color set with SetCustomAccent(), SetCustomDropdownSelected(), SetCustomProperty(), etc. is automatically included
  • Topbar and All UI Elements: Custom colors for window topbar (title, author, icon, background), tabs, elements, popups, dialogs, and all other UI components are automatically saved and restored

Troubleshooting: Custom Colors Not Saving/Loading

If your custom theme colors aren't being saved or loaded correctly, check the following:

  • Ensure colors are set before saving: Make sure you've called WindUI:SetCustomAccent(), WindUI:SetCustomWindowTopbarBackground(), etc. before calling configFile:Save()
  • Check the console: The config system prints debug messages showing how many custom theme overrides are being saved/loaded. If you see "CustomOverrides table is empty", it means no custom colors were set before saving
  • Verify the config file: Check the saved config JSON file - it should have a __themeOverrides section with your custom colors
  • Load timing: Custom colors are applied immediately when loading a config, but if you have UI elements created after loading, they may need a moment to register. The system automatically updates the theme again after a short delay to catch late-registered elements
-- ✅ CORRECT: Set colors BEFORE saving
WindUI:SetCustomAccent(Color3.fromHex("#30ff6a"))
WindUI:SetCustomWindowTopbarBackground(Color3.fromHex("#0f0f0f"))
WindUI:SetCustomBackground(Color3.fromHex("#1a1a1a"))

-- Now save - all colors will be included
local configFile = ConfigManager:CreateConfig("myconfig")
configFile:Save()

-- ❌ WRONG: Setting colors AFTER saving won't work
-- configFile:Save()
-- WindUI:SetCustomAccent(Color3.fromHex("#30ff6a")) -- Too late!

Example: Complete Theme Customization Workflow

-- User customizes their UI colors
WindUI:SetCustomAccent(Color3.fromHex("#30ff6a"))
WindUI:SetCustomBackground(Color3.fromHex("#1a1a1a"))

-- ✅ All element values are restored!
-- ✅ UI looks exactly as it did when saved!

Config Methods Reference

Method Description
ConfigManager:CreateConfig(name) Creates a new config module with the specified name (or gets existing one from memory). Returns a config object with Save(), Load(), Register(), Set(), Get() methods. Sets the config as current and auto-registers pending flags. Recommended for overwriting configs - works even if config isn't loaded in memory. Note: If a config with this name already exists, saving will overwrite it.
ConfigManager:LoadConfigByName(name) Loads a config by name. Returns success (boolean) and result. Automatically creates the config if it doesn't exist.
ConfigManager:AllConfigs() Returns an array of all available config names.
ConfigManager:ConfigExists(name) Checks if a config with the specified name exists. Returns boolean.
ConfigManager:GetConfig(name) Gets an existing config module by name from memory. Returns the config object or nil if not in memory. Note: For overwriting configs, use CreateConfig() instead, as it works even if the config isn't loaded in memory.
ConfigManager:DeleteConfig(name) Deletes a config by name. Returns success (boolean) and message.
configFile:Save() Saves the config. Returns (success, errorMessage). Automatically includes all registered UI elements and custom theme colors. Overwrites existing configs with the same name.
configFile:Load() Loads the config. Automatically restores all registered UI elements and custom theme colors.
configFile:Register(flag, element) Registers a UI element with the config using its Flag name. The element's value will be saved/loaded.
configFile:Set(key, value) Saves custom data to the config. Can store any serializable value.
configFile:Get(key) Retrieves custom data from the config by key.

Important: Using Flag Parameter for Toggles and Dropdowns

Critical: For toggles, dropdowns, and other UI elements to have their states (on/off, selected values) saved and loaded properly, you must include a Flag parameter when creating them. Without the Flag parameter, the config system cannot track and restore their states.

-- ❌ WRONG - Toggle without Flag parameter
local myToggle = MainTab:Toggle({
    Title = "AutoFarm",
    Default = false,
    Callback = function(value)
        -- State changes won't be saved/loaded!
    end
})

-- ✅ CORRECT - Toggle with Flag parameter
local myToggle = MainTab:Toggle({
    Title = "AutoFarm",
    Default = false,
    Flag = "AutoFarmToggle", -- Required for saving/loading state!
    Callback = function(value)
        -- State will be saved and restored correctly
    end
})

-- ❌ WRONG - Dropdown without Flag parameter
local myDropdown = MainTab:Dropdown({
    Title = "Auto Tap Mode",
    Values = {{Title = "Nearest"}, {Title = "Random"}},
    Value = "Nearest",
    Callback = function(option)
        -- Selected value won't be saved/loaded!
    end
})

-- ✅ CORRECT - Dropdown with Flag parameter
local myDropdown = MainTab:Dropdown({
    Title = "Auto Tap Mode",
    Values = {{Title = "Nearest"}, {Title = "Random"}},
    Value = "Nearest",
    Flag = "AutoTapModeDropdown", -- Required for saving/loading selection!
    Callback = function(option)
        -- Selected value will be saved and restored correctly
    end
})

-- Now when you save a config, these elements' states are automatically included
local configFile = ConfigManager:CreateConfig("myconfig")
configFile:Save() -- ✅ Toggle on/off state and dropdown selection are saved!

-- When you load the config, states are automatically restored
ConfigManager:LoadConfigByName("myconfig") -- ✅ Toggle and dropdown restore to saved states!

Key Points:

  • Always use Flag parameter: Every toggle, dropdown, slider, input, keybind, and colorpicker should have a unique Flag name
  • Flag names must be unique: Each element needs a different flag name (e.g., "AutoFarmToggle", "AutoTapToggle", "AutoTapModeDropdown")
  • Auto-registration: Elements with Flag are automatically registered with the config system - no manual registration needed
  • State persistence: With Flag, toggle on/off states and dropdown selections are automatically saved and restored
  • Without Flag: Elements without Flag won't have their states saved, even if you manually register them

Troubleshooting: Toggles/Dropdowns Not Saving States

If your toggles and dropdowns aren't saving their states when you save a config, check the following:

  1. Verify Flag parameter exists: Make sure each toggle/dropdown has a Flag = "UniqueName" parameter
  2. Check Flag names are unique: Each element must have a different flag name
  3. Ensure config is saved properly: Check that configFile:Save() returns true
  4. Verify elements are registered: Elements with Flag are auto-registered, but you can verify by checking Window.PendingFlags

What Gets Saved Automatically

When you save a config, the following are automatically included:

  • All registered UI elements: Toggles, Sliders, Dropdowns, Inputs, Keybinds, Colorpickers (values are saved) - only if they have Flag parameter
  • All custom theme colors: Accent, Background, Text, Button, Icon, Dialog, Outline, Hover, Placeholder, DropdownSelected, and all other custom properties
  • Custom data: Any data you save using configFile:Set(key, value)

Config File Location

Configs are saved in: {Folder}/config/{configName}.json

For example, if your folder is "myscript" and config name is "default", the file will be saved at: myscript/config/default.json

Note: Saving a config with the same name will overwrite the existing file at that location. The old config data will be completely replaced.
